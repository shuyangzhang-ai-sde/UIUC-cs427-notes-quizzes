14.1 Learning Objectives
Software Maintenance4 TypesAdaptiveCorrectiveTesting
  • Identify the four software maintenance types and avoid common distractors.
  • Distinguish adaptive from corrective maintenance scenarios.
  • Explain why corrective maintenance needs reproduce-fix-regression testing.
From Release to Maintenance
Post-releaseEvolutionSDLCMaintenance
TopicWhat to remember
DefinitionSoftware maintenance means modifying software after release to keep it correct, compatible, and maintainable over time.
ScopeIncludes bug fixes, environment adaptation, feature/performance improvements, and prevention-oriented cleanup.
Exam angleThe official four types are Corrective, Adaptive, Perfective, Preventive; "Proactive" is a distractor label.
Program maintenance
The maintenance stage is not just bug fixing - it is continuous software evolution after deployment.
The Four Maintenance Types
CorrectiveAdaptivePerfectivePreventive
TypePrimary goalTypical trigger
CorrectiveFix defects/failuresBug report, crash, wrong output
AdaptiveKeep compatibilityOS/API/hardware/regulation changes
PerfectiveImprove functionality/performanceUser or product enhancement requests
PreventiveReduce future risk/debtMaintainability and reliability concerns
Exam Tip
Proactive is not one of the official four maintenance types. The four are Corrective, Adaptive, Perfective, and Preventive.
Adaptive vs Corrective + Testing Needs
QuestionCorrectiveAdaptive
What changed?Code had a defectExternal environment changed
Was software previously correct?No (fault existed)Usually yes, now incompatible
ExampleNull-pointer crash fixUpdate for new OS or API version
True / False Focus
"Adaptive maintenance is performed to adjust software to a changed or changing environment." is True.
  • Reproduce the original failure in a controlled setup.
  • Apply the fix and verify the specific issue is resolved.
  • Run regression checks to ensure no new defects were introduced.
Exam Trap
Corrective maintenance testing is not "new feature only" testing and should never be skipped just because a change appears minor.

Size MetricsComplexity MetricsLOCCyclomatic Complexity
  • Distinguish size metrics from complexity metrics.
  • Explain why Cyclomatic Complexity is not a size metric.
  • Interpret CC thresholds, especially why CC > 10 signals high testing risk.
Size vs Complexity Metrics
LOC#Methods#ClassesCC
MetricCategoryWhat it captures
Lines of Code (LOC)SizeAmount of source lines
Number of MethodsSizeHow many methods/functions are defined
Number of ClassesSizeHow many class units exist
Cyclomatic ComplexityComplexityNumber of independent control-flow paths
Exam tip
Cyclomatic Complexity is NOT a size metric; it measures logical branching complexity.
LOC Limitations and CC Thresholds
StatementVerdictReason
LOC is always reliable for quality/complexityFalseCode can be long but simple, or short but hard to understand.
Method with CC > 10 is hard to test and fault-proneTrueMore branches increase paths and test burden.
  • LOC is useful for rough sizing, but not sufficient for quality judgment.
  • CC approximates independent execution paths and test difficulty.
  • Refactor high-CC methods to reduce risk and improve testability.
Quick memory aid
Size metrics tell how much code you have; complexity metrics tell how hard that code is to reason about and test.
14.3 Learning Objectives
WMCDITCBONOC
  • Understand what each OO metric (WMC, DIT, CBO, NOC) measures.
  • Explain how high vs. low values of each metric affect complexity, testability, and reusability.
  • Identify which metric is described given a definition or consequence.
OO Metrics — Overview
WMCDITCBONOCCK Suite
MetricFull NameWhat it Measures
WMCWeighted Methods Per ClassSum of complexities of all methods in a class
DITDepth of Inheritance TreeHow deep a class is in the inheritance hierarchy
CBOCoupling Between Object ClassesNumber of other classes a class is coupled to
NOCNumber of ChildrenNumber of direct (immediate) subclasses
Core Principle
  • Lower is better for WMC, DIT, and CBO — simpler, less coupled, more testable
  • NOC is more nuanced — high NOC means wider reuse but requires more careful design
  • All four belong to the Chidamber & Kemerer (CK) Metrics Suite
WMC — Weighted Methods Per Class
WMCMethod ComplexityCyclomatic Complexity
AspectDetail
FormulaWMC = Σ complexity(method_i) for all methods in class
Uniform weightIf each method has weight 1, WMC = total number of methods
High WMCHarder to understand, test, maintain, and reuse the class
Low WMCSimpler class; more likely to be reusable and testable
Exam Point
WMC is the metric that measures the sum of complexities of methods in a class — not DIT, CBO, or NOC.
DIT — Depth of Inheritance Tree
DITInheritanceReusePredictability
DIT ValueImplication
Low DITFewer inherited methods; behavior is easier to predict and understand
High DITMore inherited methods → more reuse, but harder to predict behavior and more complex
  • DIT applies to both single and multiple inheritance hierarchies
  • Reuse ↑ as DIT increases; Predictability ↓ as DIT increases
Common Misconception
Higher DIT does NOT make behavior easier to predict — more inherited methods = harder-to-predict behavior.
Exam Point
Higher DIT → more reuse but harder to predict + more complex.
CBO — Coupling Between Object Classes
CBOCouplingTestabilityReusability
CBO LevelEffect on TestabilityEffect on Reusability
Low CBOEasier to testMore reusable
High CBOHarder to testLess reusable
  • High CBO means many dependencies → changes elsewhere can break this class
  • Goal: low coupling, high cohesion — fundamental OO design principle
Exam Point — False Statement Alert
"Higher CBO makes a class easier to test and more reusable" is FALSE. Higher CBO does the opposite.
NOC — Number of Children
NOCSubclassesInheritance Fan-out
NOC LevelImplication
High NOCClass is widely reused via inheritance; changes impact many subclasses — more testing needed
Low NOCLimited reuse through inheritance; less ripple effect from changes
DIT vs. NOC
DIT = how deep a class is (ancestors above). NOC = how wide its influence is (children below). Both are in the CK metrics suite.
14.4 Learning Objectives
RefactoringExtract MethodWhen to Refactor
  • Explain what refactoring is and what it is not.
  • Identify whether a given activity qualifies as refactoring.
  • Explain when refactoring should (and should not) be combined with behavior changes.
  • Describe the benefits and one trap of the Extract Method refactoring technique.
What is Refactoring?
RefactoringInternal StructureExternal BehaviorCode Quality
Refactoring is the process of changing the internal structure of a program without changing its external behavior. The observable functionality remains identical — only the code's organization, readability, or design improves.
AspectRefactoringNOT Refactoring
External behaviorUnchangedModified / new features added
Internal structureImprovedUntouched
PurposeImprove code qualityAdd features / fix bugs
TestsShould still passNew tests may be needed
ActivityChanges Behavior?Is Refactoring?
Reorganize code structureNoYes
Add a new featureYesNo
Fix a bugYes (corrects behavior)No
Optimize performance (same output)NoYes (if behavior unchanged)
Optimize but alter functionalityYesNo
Exam Tip
The key distinguisher: refactoring never alters external behavior. Options mentioning "adding new features" or "altering functionality" are always wrong answers for refactoring.
When to Refactor — The Golden Rule
Separation of ConcernsBehavior ChangeCommit Discipline
Refactoring should NOT be done at the same time as changing behavior. These are two separate activities and should be performed in separate steps (ideally separate commits).
  • Mixing refactoring with behavior change makes it hard to isolate bugs
  • If something breaks, you won't know if it was the refactor or the new logic
  • Best practice: refactor first (or after), then change behavior in a distinct step
  • Each step should be independently verifiable by running tests
StatementVerdict
"Refactoring should always be done after changing behavior"FALSE
"Refactoring and behavior change should be kept separate"TRUE
"Refactoring can be done before or after, but not simultaneously"TRUE
One-Sentence Summary
Refactoring is its own activity — never mix it with behavior changes; always keep them in separate steps.
Extract Method
Extract MethodReusabilityReadabilityMethod Count ↑Single Responsibility
Extract Method takes a code fragment that can be grouped together and moves it into a new method with a name that explains its purpose.
BenefitExplanationReal Benefit?
Increases reusabilityExtracted method can be called from multiple placesYes
Increases readabilityCode reads like well-named function calls instead of inline logicYes
Makes purpose clearerA good method name documents the intent of the codeYes
Decreases number of methodsExtraction adds a new method — count increases, not decreasesNo — this is the trap!
  • Each extraction creates one new method — overall method count goes up, not down
  • Supports the Single Responsibility Principle — each method does one thing
  • Easier to unit-test smaller, extracted methods independently
Exam Tip
"Decreases the number of methods in a class" is the trap answer — Extract Method always increases method count. Reusability, readability, and understandability are all genuine benefits.
Extract Method — Before & After
BeforeAfter Extract Method
void printOrder() { // print header System.out.println("----"); System.out.println("Order"); System.out.println("----"); // print items for (Item i : items) System.out.println(i); }
void printOrder() { printHeader(); printItems(); } void printHeader() { System.out.println("----"); System.out.println("Order"); System.out.println("----"); } void printItems() { for (Item i : items) System.out.println(i); }
Key Observation
External behavior (what gets printed) is identical. Internal structure is clearer. Method count went from 1 → 3. This is the essence of Extract Method refactoring.
14.5 Learning Objectives
Introduce Parameter ObjectRefactoring StepsExtract MethodBenefits
  • Identify the correct first step of Introduce Parameter Object refactoring.
  • Apply the ordered process: create class, add parameter object, then remove old parameters incrementally.
  • Explain why follow-up extraction into the new class improves cohesion and readability.
Introduce Parameter Object — Example
Data ClumpsParameter ObjectCohesion
When related parameters repeatedly appear together, create a dedicated object to represent them. This reduces long argument lists and makes intent explicit at call sites.
BeforeAfter
book(startDate, endDate, startTime, endTime)book(dateTimeRange)
check(startDate, endDate, startTime, endTime)check(dateTimeRange)
Data clumps and order mistakes are common.Single named object improves clarity and maintainability.
Exam Tip
The first step is to make a new class for the parameter group before adding it to signatures.
Recommended Refactoring Sequence
Order MattersIncrementalSafety
StepAction
1Create the new parameter object class with fields/constructor.
2Add a new method parameter typed as that class.
3Update call sites to pass the new object.
4Remove original parameters one-by-one and replace usage with object fields.
5Move related logic into the new class when appropriate (Extract Method follow-up).
Follow-up
After introducing the parameter object, check if behavior should move with data. This often improves object-oriented cohesion.
Benefits and Common Trap
ReadabilityMaintainabilityTrap Option
  • Reduces parameters in method calls.
  • Groups related values into one conceptually meaningful object.
  • Improves readability and future maintenance.
Quiz Trap
“Makes code more difficult to understand” is not a benefit.
14.6 Learning Objectives
Code SmellMaintainabilityDuplicated CodeLong Method
  • Identify what a code smell indicates about software design quality.
  • Connect common smells to practical maintenance risks in real codebases.
  • Select suitable refactorings to improve readability and long-term changeability.
What Is a Code Smell?
Design DecayWarning SignNot Always a Bug
A code smell is a warning sign that the code's internal structure is becoming harder to evolve. It does not always break behavior immediately, but it raises the risk and cost of future changes.
OptionIs it a code smell?Why
Design starts to decayYesSmells point to maintainability and design quality issues.
Syntax errorNoSyntax errors are immediate compile-time defects, not smells.
Optimized codeNoOptimization is not itself a warning sign.
Security vulnerabilityNo (different category)Security issues are vulnerabilities, not smell classification.
Exam Tip
The best definition is: a clear sign that code design is starting to decay.
Duplicated Code and Refactoring
Duplicated CodeExtract MethodSingle Source of Truth
When similar logic appears in multiple places, maintenance becomes error-prone. The most common first fix is Extract Method: move the repeated fragment into one named method and call it from each location.
RefactoringBest for duplicated code?Reason
Extract MethodYesConsolidates repeated logic into one reusable implementation.
Introduce Parameter ObjectSometimesSolves long parameter lists / data clumps, not primary duplication cleanup.
Inline MethodUsually noCan increase duplication by spreading logic back into callers.
Rename VariableNoImproves naming only; does not remove duplicated logic.
Long Method Smell
Long MethodComplexityReadability
  • Long methods mix too many responsibilities and hide intent.
  • They increase cognitive load and slow down bug localization.
  • They are harder to unit test because setup and branching become tangled.
  • Refactor by extracting cohesive chunks with names that express purpose.
True/False Focus
The statement is True: a Long Method smell is problematic because complexity makes code harder to understand and maintain.
Quick Review
  • Code smell = design quality warning, not necessarily immediate failure.
  • Duplicated Code is commonly addressed with Extract Method.
  • Long Method is a maintainability risk and should be decomposed.
