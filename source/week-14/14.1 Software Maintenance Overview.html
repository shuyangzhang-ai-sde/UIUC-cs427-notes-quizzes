Definition
What is Software Maintenance?
Software Maintenance Post-release Evolution SDLC
Software maintenance is the process of modifying and updating software after its initial release. It is the final — and often longest — phase of the SDLC, encompassing all work done to keep software functional, relevant, and reliable over time.
  • Maintenance typically accounts for 40–80% of total software cost over a system's lifetime
  • It is not limited to fixing bugs — it also covers adaptation, improvement, and prevention
  • Every change made post-release falls into one of the four maintenance categories
One-line Summary
Software maintenance keeps software alive and useful after deployment — it is ongoing, not a one-time event.

Exam Alert
"Proactive" is NOT one of the four types. The four types are: Corrective, Adaptive, Perfective, and Preventive. "Proactive" is a common distractor — don't confuse it with Preventive maintenance.

Type: Corrective Maintenance
Fixing Bugs and Failures
Corrective Bug Fix Defect Failure Reproduction Regression Testing
Corrective maintenance addresses defects and failures discovered after release. It is reactive — the software has broken or produced incorrect output, and the team must diagnose and repair it.
  • Step 1 — Reproduce: Replicate the failure in a controlled environment to understand it
  • Step 2 — Diagnose: Trace the fault to its root cause in the code
  • Step 3 — Fix: Apply the code change to correct the defect
  • Step 4 — Confirm: Verify the fix resolves the original failure
  • Step 5 — Regression test: Ensure no new issues were introduced by the fix
Key Exam Point — Testing for Corrective Maintenance
Testing for corrective maintenance must: (1) reproduce the failure, (2) confirm the fix works, and (3) ensure no new issues were introduced (regression testing). It is NOT limited to new features, UI testing, or skipped for minor changes.

One-line Summary
Corrective maintenance testing = reproduce → fix → verify → regression check. All three steps are mandatory.
Type: Adaptive Maintenance
Adjusting to a Changing Environment
Adaptive Environment Change OS Update Hardware Third-party API Compatibility
Adaptive maintenance modifies software to keep it functional when external conditions change. The software itself has no bug — the surrounding environment has shifted, requiring the software to be updated to remain compatible.
Environment ChangeExample Adaptation
New operating system version Update system calls or deprecated APIs
New hardware platform Recompile or adjust drivers/interfaces
Third-party service API change Update integration code to new API spec
New legal/regulatory requirements Add data privacy compliance features
Database migration Update queries/schemas for new DB version
Key Exam Point — True Statement
"Adaptive maintenance is performed to adjust software to a changed or changing environment." — This is TRUE. The key word is environment: the software adapts to external changes, not internal defects.
Distinguishing Adaptive vs. Corrective
Was it the Software or the Environment?
The critical question: did something break because of a bug already in the code, or because something outside the software changed?

Memory Tip
Adaptive = Adjust. The software adapts to its environment like an organism adapting to climate change — nothing was wrong before, the world around it changed.
