Code metrics quantify properties of source code to help engineers assess quality, predict fault-proneness, and guide refactoring decisions. They fall into two broad categories: size metrics and complexity metrics.
Size Metrics
How large is the code?
Complexity Metrics
How hard is the code to understand & test?
Key Distinction
Cyclomatic Complexity is NOT a size metric. It measures structural complexity (control flow), while LOC, number of methods, and number of classes measure size.

Size Metric
Lines of Code (LOC)
LOC SLOC Size Limitations
LOC (or SLOC — Source Lines of Code) is the most straightforward size metric: simply count the number of lines in a source file. It is easy to collect automatically, but comes with significant limitations.
  • Easy to measure automatically with tooling
  • Useful as a rough proxy for project scope or effort estimation
  • Can be compared across versions of the same codebase to track growth
Limitations
  • More lines ≠ better code — verbose styles inflate LOC without adding value
  • A skilled refactor can cut LOC in half while improving quality
  • Does not capture code structure, readability, or logic complexity
  • Different languages have very different LOC for equivalent functionality
Quiz Trap
LOC is NOT always a reliable indicator of quality or complexity. High LOC can mean redundant code; low LOC can mean unreadable one-liners. Always consider LOC alongside other metrics.

Complexity Metric
Cyclomatic Complexity
Cyclomatic Complexity Control Flow Independent Paths Fault-Prone Testability
Proposed by Thomas McCabe (1976), Cyclomatic Complexity (CC) measures the number of linearly independent paths through a method's control flow graph. It directly reflects how many different execution paths a tester must cover.
Formula: CC = E − N + 2P where E = edges, N = nodes, P = connected components (usually 1 method → P=1).
Practically: CC ≈ number of decision points (if, else if, while, for, case, &&, ||, ?) + 1.
Complexity Thresholds
1 – 5
Simple, easy to test ✓
6 – 10
Moderate complexity
> 10
Hard to test, fault-prone ✗
Quiz Point
A method with CC > 10 is considered hard to test and fault-prone. This threshold is widely accepted in industry and academia (McCabe's original guideline).
One-Sentence Summary
Cyclomatic Complexity tells you the minimum number of test cases needed to cover every branch in a method — higher CC = more tests needed = higher risk.
示例
Calculating CC — Quick Example
if else while Decision Points
Count every branching keyword as one decision point, then add 1:
  • A straight-line method with no branches → CC = 1
  • One if statement → CC = 2
  • One if + one while → CC = 3
  • Ten nested if / else if → CC = 11 → refactor recommended
Why It Matters
High CC methods are hard to reason about, hard to test completely, and statistically more likely to contain bugs. Refactoring (Extract Method, Replace Conditional with Polymorphism) can reduce CC.

Size Metrics
Definition
Quantify the volume or scale of source code — how large the codebase is in terms of countable structural units. Collected by scanning, not by reading logic.
Examples
  • LOC — total lines of source code
  • Number of Methods — functions defined per class
  • Number of Classes — class definitions in codebase

"How big is the code?"

vs
Complexity Metrics
Definition
Measure the structural complexity of code logic — how many independent execution paths exist. Requires reading the control flow, not just counting lines.
Examples
  • Cyclomatic Complexity (CC) — independent paths through a method
  • Formula: CC = E − N + 2P (edges − nodes + 2 × components)
  • Practical: decision points + 1 (if, while, for, case, &&, ||, ?)
Key Question
"How hard is the code to test & understand?"
Threshold: CC > 10 = hard to test and fault-prone.
Size metrics — found by counting (scan the code)
Complexity metrics — found by tracing logic (read the control flow)

场景对比
When to Use Each Metric — Scenarios & Applications
LOC Methods Classes Cyclomatic Complexity
Size Lines of Code (LOC)
Most Useful When
  • Tracking codebase growth across releases or sprints
  • Making rough effort / cost estimates early in a project
  • Comparing two versions of the same file for bloat detection
  • Setting coarse size budgets for modules or services
Specific Scenarios
  • A PR adds 2 000 LOC — flag for extra review before merge
  • A "refactor" reduces LOC by 40% — sign of deduplication success
  • Comparing LOC between teams using the same language for productivity baselines
Avoid: using LOC as a quality or complexity proxy — more lines ≠ better or harder code.
Size Number of Methods
Most Useful When
  • Evaluating whether a class has too many responsibilities (Single Responsibility Principle)
  • Detecting God Objects — classes with dozens of methods doing unrelated work
  • Measuring the public API surface exposed by a module
  • Guiding Extract Class refactoring decisions
Specific Scenarios
  • A class has 50 methods → strong signal to split into cohesive sub-classes
  • Comparing method counts before/after a refactoring sprint to verify decomposition
  • Setting a per-class method limit (e.g., ≤ 20) as a team coding standard
Note: measures breadth of a class, not the internal complexity of each method.
Size Number of Classes
Most Useful When
  • Assessing overall modularity and design decomposition of the system
  • Tracking architectural growth trends as features are added
  • Comparing design alternatives — more classes can mean better separation of concerns or unnecessary fragmentation
  • Supporting coupling / cohesion analysis alongside dependency metrics
Specific Scenarios
  • A monolith has 3 classes → flag for poor decomposition; hard to test in isolation
  • Microservice grows from 10 → 80 classes in one sprint → review for over-engineering
  • Used with CBO (Coupling Between Objects) to spot over-connected class webs
Note: works best alongside cohesion/coupling metrics, not in isolation.
Complexity Cyclomatic Complexity (CC)
Most Useful When
  • Planning test coverage — CC gives the minimum branch test cases needed
  • Identifying methods that are high-risk for bugs (CC > 10)
  • Prioritizing refactoring targets before a release or code review
  • Enforcing a team complexity budget in CI/CD pipelines (fail build if CC > threshold)
Specific Scenarios
  • A payment-processing method has CC = 18 → high bug risk; decompose with Extract Method
  • Security-critical authentication logic: enforce CC ≤ 5 for full branch coverage confidence
  • Legacy code audit: rank all methods by CC to build a tech-debt prioritization list
Threshold: 1–5 simple ✓  |  6–10 moderate  |  > 10 fault-prone ✗
考点总结
Exam Traps — Know These Cold
Common Mistakes Quiz Focus
  • Cyclomatic Complexity is NOT a size metric — it is a structural complexity metric. LOC, # methods, # classes are size metrics.
  • LOC is NOT always reliable for measuring quality or complexity — more lines can mean verbosity, not value.
  • CC > 10 is the threshold for "hard to test and fault-prone" — this is a standard industry guideline, not just a suggestion.
Quick Memory Aid
Size metrics = What you can count by scanning (lines, methods, classes). Complexity metric = What you can only find by reading the logic (branches, paths).
