Core Concept
What is Refactoring?
Refactoring Internal Structure External Behavior Code Quality
Refactoring is the process of changing the internal structure of a program without changing its external behavior. The observable functionality remains identical — only the code's organization, readability, or design improves.
Exam Tip
The key distinguisher: refactoring never alters external behavior. Options mentioning "adding new features" or "altering functionality" are always wrong answers for refactoring.

Timing Rule
Refactoring and Behavior Change: The Golden Rule
Separation of Concerns Behavior Change Commit Discipline
Refactoring should NOT be done at the same time as changing behavior. These are two separate activities and should be performed in separate steps (ideally separate commits).
  • Mixing refactoring with behavior change makes it hard to isolate bugs
  • If something breaks, you won't know if it was the refactor or the new logic
  • Best practice: refactor first (or after), then change behavior in a distinct step
  • Each step should be independently verifiable by running tests
StatementVerdict
"Refactoring should always be done after changing behavior" FALSE
"Refactoring and behavior change should be kept separate" TRUE
"Refactoring can be done before or after a behavior change, but not simultaneously" TRUE
One-Sentence Summary
Refactoring is its own activity — never mix it with behavior changes; always keep them in separate steps.
Refactoring Technique
Extract Method
Extract Method Reusability Readability Method Count Single Responsibility
Extract Method takes a code fragment that can be grouped together and moves it into a new method with a name that explains its purpose. This is one of the most commonly used refactoring techniques.
BenefitExplanationReal Benefit?
Increases reusability Extracted method can be called from multiple places ✓ Yes
Increases readability Code reads like well-named function calls instead of inline logic ✓ Yes
Makes purpose clearer A good method name documents the intent of the code ✓ Yes
Decreases number of methods Extraction adds a new method — method count increases, not decreases ✗ No — this is the trap!
  • Each extraction creates one new method — overall method count goes up, not down
  • The tradeoff is worth it: more methods, but each is shorter, named, and focused
  • Supports the Single Responsibility Principle — each method does one thing
  • Easier to unit-test smaller, extracted methods independently
Example
Before vs. After Extract Method
A practical illustration of how Extract Method transforms code structure:
BeforeAfter Extract Method
void printOrder() { // print header System.out.println("----"); System.out.println("Order"); System.out.println("----");
// print items for (Item i : items) System.out.println(i); }		 void printOrder() { printHeader(); printItems(); } void printHeader() { System.out.println("----"); System.out.println("Order"); System.out.println("----"); } void printItems() { for (Item i : items) System.out.println(i); }
Key Observation
External behavior (what gets printed) is identical. Internal structure is clearer. Method count went from 1 → 3. This is the essence of Extract Method refactoring.
