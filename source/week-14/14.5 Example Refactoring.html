Chapter 14.5 — Example Refactoring

Concept
Introduce Parameter Object
Introduce Parameter Object Refactoring Parameter Group New Class Data Clumps
A refactoring technique that groups a set of related parameters that frequently appear together across multiple method signatures into a single new class. Instead of passing several individual arguments, callers pass one object that encapsulates all of them.
Exam Tip
The first step is always to make a new class for the parameter group — before touching any existing method signatures or call sites.
Motivation
Why Use It? — Data Clumps
Data Clumps Code Smell Long Parameter List
When you see the same group of parameters appearing together in multiple places, this is a code smell called Data Clumps. These parameters belong together conceptually, and creating a class for them clarifies their relationship.
Before (Data Clump)After (Parameter Object)
book(startDate, endDate, startTime, endTime) book(dateTimeRange)
check(startDate, endDate, startTime, endTime) check(dateTimeRange)
Multiple scattered parameters, easy to mis-order Single cohesive object, named intent
Procedure
Steps to Perform Introduce Parameter Object
Step-by-Step New Class New Parameter Incremental Removal
  1. 1 Make a new class for the group of parameters. This is the parameter object class. Add fields and a constructor matching the parameters you want to group.
  2. 2 Add a new parameter to the method whose type is the new class, while keeping all original parameters for now.
  3. 3 Update all call sites to pass in the new parameter object (set to null initially if needed), ensuring nothing breaks yet.
  4. 4 Incrementally remove each original parameter, replacing its usage inside the method body with accesses to the new object. Update all call sites after each removal.
  5. 5 Check if any logic can be moved into methods of the new class — this may lead to an Extract Method refactoring on the parameter object itself.
Key Order
Step 1 (create class) → Step 2 (add new param) → Step 4 (remove old params one by one). Never remove before adding the replacement.
Order Trap
Common Mistake — Wrong First Step
Wrong Order First Step
OptionCorrect First Step?Why
Make a new class ✓ Yes You need the type before you can add it as a parameter
Add a new parameter ✗ No Can't add a typed parameter before the class exists
Remove original parameters ✗ No Removing before replacement breaks callers immediately
Replace calls with inlined code ✗ No That is Extract Method logic, not the start of this refactoring

NOT a Benefit — Exam Trap
"Makes the code more difficult to understand" is listed as a distractor option. This is false — Introduce Parameter Object improves readability, it does not hinder it.
True/False Statement
After Refactoring: Should You Check for Extract Method?
TRUE Extract Method Follow-up Step
TRUE. After introducing a parameter object, you should review whether any logic scattered across call sites or methods can be moved into the new class as its own methods. This follow-up step is called Extract Method.
Why This Makes Sense
Once related data lives in one class, behavior that operates on that data is a natural fit to live there too — this is a step toward better object-oriented cohesion and is the spirit of "Tell, Don't Ask."
Related Refactorings
Extract Method
Extract Method Move Behavior Cohesion
Extract Method takes a fragment of code from a method body and places it in a new, well-named method. In the context of Introduce Parameter Object, it means moving logic that works on the parameter object's data into the parameter object class itself.
  • Improves readability by giving a name to a block of logic
  • Reduces method length
  • Encourages placing behavior next to the data it operates on
  • Often a natural follow-up to Introduce Parameter Object
Comparison
Introduce Parameter Object vs. Related Patterns
TechniqueWhat It DoesRelationship
Introduce Parameter Object Groups related parameters into a new class The primary technique
Extract Method Moves code fragments into a new method Follow-up: move behavior into the new class
Replace Inline Code with Method Call The reverse of Extract Method — consolidates duplicated inline logic Distinct; not the same as Introduce Parameter Object
Preserve Whole Object Pass an entire object instead of extracting several values from it Similar intent; avoids destructuring existing objects
One-line Summary
Introduce Parameter Object = create a class → add it as parameter → incrementally remove old params → optionally Extract Method to move behavior into the new class.
