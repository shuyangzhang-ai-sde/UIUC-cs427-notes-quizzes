JUnit Framework
Slide content
  • Open source Java unit testing framework (junit.org) for writing and running repeatable automated tests
  • The latest version is JUnit 5, requiring Java 8 (or higher) at runtime
    • can still test code compiled with previous JDK versions
  • Main JUnit features
    • Assertions for checking expected results
    • Assumptions for filtering out tests
    • Annotations for configuring tests and extending the framework
      • disabling tests
      • conditional test execution
      • displaying names, tagging and filtering
      • …
    • Test suites for organizing and running tests
    • Console launcher/test engines for executing tests

(JUnit, 2018)

Transcript

The JUnit framework is an open source Java unit testing framework for the work as write and run repeatable automated tests. Currently, the latest version of JUnit is JUnit 5, which requires Java 8 or higher at runtime. However, you can still use JUnit 5 to test the code under test compiled with previous JDK versions, not necessary Java 8 or higher.

There are a number of main features of JUnit. For example, you can write different kind of assertions for checking expected results.

You can also write assumptions for filtering out certain tests especially when the input values are being passed in to the tests such as a praumatize test which we will discuss in later lecture videos. There are also a number of other annotations for configuring tests and expanding the framework. For example, you could use annotations to tentatively disable certain tests and you can also configure certain conditional test execution. For example, you can configure certain unit tests to be executed when you are using a specific version of the operating system or using a specific version of the Java JDK version. You can use annotation to display names of the unit test when you are running these tests and reporting the failures or the passing status of these tests. You can also tag different unit tests and filter out the executions of certain unit tests viz a specific tag. You can group certain unit test cases into test suites so that you could run specific test suites.

In order to automatically execute certain unit tests, you can leverage console launcher or some tests engines being integrated into different kind of IDEs as a graphical user interface for running these unit tests.

Example JUnit Test Case
Slide content

Class under test: java.util.Stack

Test classes/methods don't need to be public.

import java.util.Stack; import org.junit.jupiter.api.*; import static org.junit.jupiter.api.Assertions.*; class TestingAStackDemo { @Test @DisplayName("Push an element into empty stack") void testPushOneElem() { String element1 = "element1"; Stack<Object> stack = new Stack<>(); stack.push(element1); assertEquals(element1, stack.peek()); assertFalse(stack.isEmpty()); } }

(JUnit Example, 2018)

Transcript

Now let's take a look at the example with JUnit test case. The class under test is the stack class from the Java util package. In order to test such stack class, we need to import such class so that our unit test can have access to this class. In addition, we also import these particular package from the JUnit framework. These particular package includes those typical annotations for unit tests such as @Test, @DisplayName. In addition, in order to allow us to write different kinds of assertions in our unit test, we also need to import this particular package including the definitions of different kinds of assertions.

Then we have the test class we named as TestingAStackDemo. Then within this test class, we can define different unit test cases as test methods. Here we shall only one example of the unit test as testPushOneElement. Note that the test method have void return type. In order to allow the JUnit testing framework to realize that a particular method is a unit test, we can add annotation @Test. We can also add specific string which can include space as a DisplayName when you are running the tests and getting the test reports. Note that the name of the test method cannot allow you to include space but this plain name can have space there.

Then, the body of the unit test method includes a sequence of method calls along with the argument values for each method call and finally we also have assertions. Note that test classes or methods don't necessarily need to be public. We don't need to add the prefix public in front of the tests classes or methods. For this particular unit tests, basically the behavior is the following. We first create an empty stack and then we push element one as a string into the stack then we make sure top of the stack, which can be attained by invoking peek, to be the same as the element one. In addition, we also double-check that the stack is not empty. Basically, we invoke isEmpty method defined on stack and use assertion method as assertFalse.

Basically, we are asserting that the stack is empty return value is false.

BeforeEach and AfterEach
Slide content
class TestingAStackDemo { Stack<Object> stack; String element1 = "element1"; String element2 = "element2"; @BeforeEach void createNewStack() { stack = new Stack<>(); } @Test void testAddOneElem() { stack.push(element1); assertEquals(element1, stack.peek()); }
@Test void testAddTwoElems() { stack.push(element1); stack.push(element2); assertEquals(element2, stack.peek()); } @AfterEach void createNewStack() { assertFalse( stack.isEmpty()); } }

(JUnit Example, 2018)

Transcript

For a single class, or a single method from the class under test, we need to write multiple unit test cases instead of just a single unit test case. To allow us better manage multiple unit test cases for testing the same method or same class, we can structure our unit test to share certain objects being defined in our test class. For example, we can define instance object fields stack for our test class, and we can initialize two strings as the two elements that we would like to push into the stack for our later written unit test. We add a notation as at BeforeEach to indicate such methods to be executed before we execute each unit test in these test class. Basically, BeforeEach method execution would allow us to initialize the stack with an empty stack. In this way, we don't have interference across different executions of the different unit test methods.

Then we have tool unit test methods for testing the push behavior. The first one is to add one element more precisely push one element into this empty stack and then we check the peek, basically, the top of the stack would be the same as the element1 that we just pushed. For the second test, we intend to test the behaviors of the stack after we push two elements. After that, we double-check the top of the stack as the return of the peek methods defined on stack to be the same as the element2, basically, the second element that we just pushed. In addiction, we can have additional annotations as AfterEach as the name indicates, we execute the methods being annotated as AfterEach. After the execution of each unit test method, in our example, we basically would like to double-check that the stack is not empty. For this particular test class, we have such behavior as that after we execute both has methods our stack is not empty.

But in general, we may not have such behaviors across all different test methods defined on the test class. So normally, we use such BeforeEach as more like initialization and use the methods mean annotated with AfterEach to carry out some clean up on operations after the testing is done.

Anti-Pattern: Tests Combined
Slide content
class TestingAStackDemo { Stack<Object> stack; String element1 = "element1"; String element2 = "element2"; @BeforeEach void createNewStack() { stack = new Stack<>(); } @Test void testAddOneAndTwoElems() { stack.push(element1); assertEquals(element1, stack.peek()); stack.push(element2); assertEquals(element2, stack.peek()); }
@AfterEach void createNewStack() { assertFalse( stack.isEmpty()); } }
Transcript

When we are writing unit test, we need to watch out anti-pattern. Basically, we should avoid combining multiple unit tests into a single big unit test. For the earlier tool unit test, some developers may combine them into a single unit tests. Basically, after we push one element, we check whether the top of the stack is the same as the element we just push. After that, we further push the second element and we further check whether now the top of the stack would be the same as the second element that we just pushed.

So, when we are not carrying out test automation, basically automatically running this unit test, then combining different unit tests together can allow you to reduce the efforts for re-running the prefix steps of certain other unit tests. However, when we use the automatic testing frameworks such as JUnit, we can break such big unit tests into smaller unit tests. The benefit is that we can have an easier time to debug the failures once the failures happens. The second reason is that, if you write a big unit test like what we have put on the slide, then if the earlier assertion fails, then the whole test would then execute and therefore we don't get to observe whether the later assertions will fail or not, therefore limiting the information available to us for debugging the failure. In summary, we need to avoid putting these kind of like a state modifying methods. Basically, the methods that can modify the state of the class under test in-between these assertions for checking the behaviors. We can break such a big unit test into multiple unit tests as we have written in earlier slides.
