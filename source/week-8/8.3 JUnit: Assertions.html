Example JUnit Test Case
Slide content

Class under test: java.util.Stack

Test classes/methods don't need to be public.

import java.util.Stack; import org.junit.jupiter.api.*; import static org.junit.jupiter.api.Assertions.*; class TestingAStackDemo { @Test @DisplayName("Push an element into empty stack") void testPushOneElem() { String element1 = "element1"; Stack<Object> stack = new Stack<>(); stack.push(element1); assertEquals(element1, stack.peek()); assertFalse(stack.isEmpty()); } }
Transcript

Now, let's revisit the example JUnit test case. In the end of the unit test method, we have assertions using assertion API methods from the JUnit framework. There are many different kinds of assertion API methods being defined from the JUnit framework.

Assertions
Slide content

static methods defined in org.junit.jupiter.api.Assertions

  • static void assertEquals (byte expected, byte actual)
  • static void assertEquals (char expected, char actual)
  • static void assertEquals (double expected, double actual, double delta)
    • Asserts that expected and actual are equal within the given delta
  • static void assertEquals (float expected, float actual, float delta)
    • Asserts that expected and actual are equal within the given delta
  • static void assertEquals (int expected, int actual)
  • static void assertEquals (long expected, long actual)
  • static void assertEquals (short expected, short actual)
  • static void assertEquals (Object expected, Object actual)
    • Asserts that expected and actual are equal, using equals() method
  • static void assertSame (Object expected, Object actual)
    • Asserts that expected and actual refer to the same object
  • …

(JUnit Assertions, 2018)

Transcript

They are in the form of static methods defined in this particular package from the JUnit framework. For different types of values being compared for equality, we have different kinds of assertEquals API methods. For example, for bytes, character, double, float, integer, long, short, and Objects.

Let's pay attention to the assertEquals for double and float values. We also have the third argument value delta. The reason is that if we compare the equality of two double or float values, due to rounding errors, these values may have slight impositions. Therefore, just comparing these two values we may have the conclusion as not being equal. But in fact, we would like to see that they are considered as equal. Therefore, we could use this assertEquals method to assert that the expected and the actual values are equal within the given delta — the allowable difference between the two double or float values.

Let us also take a look at the last two methods, assertEquals and assertSame. Both API methods are comparing two objects, but assertEquals is comparing the expected and actual values to be equal using the equals method being defined on these two objects. Typically, we need to override the equals method being defined on the object class so that we would have some customized equality checking between two objects of a specific class.

In contrast, assertSame is comparing the reference or pointer equality of the two objects. Basically asserting that the expected and the actual objects refer to the same object. There are many more other assertions that we are not listing here. You can refer to the JUnit assertions class for the complete list of the assertions being defined in the JUnit framework.

assertAll
Slide content

static void assertAll(String heading, Executable… executables)

Asserts that all supplied executables do not throw exceptions

@Test void groupedAssertions() { // In a grouped assertion all assertions are executed, and any // failures will be reported together. assertAll("person", () -> assertEquals("John", person.getFirstName()), () -> assertEquals("Doe", person.getLastName()) ); }

(Bechtold et al., 2018)

Transcript

Next, let's take a look at the remaining assertion API methods. First, assertAll. The first argument of assertAll is a heading of a string type to be added in the beginning of the error message of assertion violations being produced by the execution engine of the JUnit framework. And then we have a list of executable code being included as the rest of the arguments of assertAll.

These assertAll API methods assert that all supplied executable code do not throw exceptions. Any of such executable code that throws an exception would not cause other executable code not to be executed. Such situation is in contrast to the normal situations where we list each of such executable code as assertions in the end of a unit test method. In that situation, an earlier assertion being violated will cause a later assertion not to be executed. But here, all the executable codes will be executed no matter whether one or more of these executable codes will throw exceptions.

For example, in the test method on the slide, we have "person" string to be added in the beginning of the error message. And then, we have these two assertEquals API method calls being executable code that we can execute to see whether any of such assertEquals throws exceptions. If so, we will add such exception-throwing messages in the final message being produced with a prefix.

assertThrows
Slide content

static <T extends Throwable> T assertThrows(Class<T> expectedType, Executable executable)

Asserts that execution of the supplied executable throws an exception of the expectedType and returns the exception.

@Test void exceptionTesting() { Throwable exception = assertThrows( IllegalArgumentException.class, () -> { throw new IllegalArgumentException("a message"); }); assertEquals("a message", exception.getMessage()); }

(Bechtold et al., 2018)

Transcript

The second assertion API method is assertThrows. This API method allows you to assert that execution of the supplied executable code throws an exception of the expected type as the first argument of this API method. And then, the exception being thrown will be returned as the return value of this API method.

Let's take a look at the example test method on the slide. We first put the expected exception type as the first argument — IllegalArgumentException. And then for the executable code we simply throw a new IllegalArgumentException. Apparently, this executable code has an exception of the type IllegalArgumentException. Then the exception is returned, being stored in the exception variable. The last statement of this test method is assertEquals. The exception's getMessage method will return the string as a message being equal to the expected result as the first argument of assertEquals.
