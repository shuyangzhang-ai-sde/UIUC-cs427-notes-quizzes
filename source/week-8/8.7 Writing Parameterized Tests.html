Separation of Concerns
Slide content

Parameterized tests enable to separate two concerns:

  • (1) The specification of external behavior (i.e., assertions)
  • (2) The generation/selection of internal test inputs (i.e., coverage)
@ParameterizedTest @ValueSource(…) void PT1(int elem, Stack<Integer> stk) { assumeTrue(stk != null); int oldSize = stk.size(); stk.push(elem); assertEquals(oldSize + 1, stk.size()); }

Besides manually writing test inputs, one can use an automatic test generation tool to construct a small test suite with high coverage!

  • E.g., AgitarOne JUnit Generator for Java programs
  • E.g., Parasoft Jtest for Java programs
  • E.g., Microsoft IntelliTest in Visual Studio for .NET programs

And then we have arrangement sometimes for bringing objects under test to a specific state or caching some values that later on we will use in the assertions. In this example, we have such arrangement to cache the values of the old size. And then we have act, basically, invoking the measure in the test. And finally, we have assert, for asserting the behavior of the core under test, would be as expected. Sometimes the assertions may also use some cache values such as oldSize is being stored in arrangement portion of the primary test.

Property-First Guideline
Slide content
  • Step 1. Classify methods to be
    • modifiers: IntSet constructor, insert, remove while modifying/constructing states
    • observers: returning some other value w/o modifying states
  • Step 2. Pairwise-combine them to formulate properties
    • Every observer o is paired with a modifier
    • Every modifier m is paired with another modifier (including m itself)
  • Step 3. Add proper assumptions to make the properties (e.g., equality, inequality) true
Transcript

Next, let me illustrate each of these guidelines one by one. In the property first guidelines, in the first step. We need to classify those methods being defined on the interface of the class and the test to be two major types. First, is modifiers, being those methods that can actually can modify the state of the receive object. Including the constructor method call, along with inset remove because these methods can modify the states of the integer set objects. Then we also classify the remaining ones to be observers. Being those that typically do not modify the receiver object stays, but can return some values to indicate the return type or inspecting the states of receiver objects.

By the way, this method should be public methods defined on the class and the test. And we don't concern about those private methods or non-private methods that we are not testing from the interface of the class and the test. In step two, we need to enumerate and those different pairwise combinations of these methods to formulate properties. For example, each observer o is paired with a modifier and each modifier m is paired with another modifier including m itself. Then in step three, after we pair them up, basically when we say we pair one methods after another. We basically invoke one method and then invoke another method after debt.
