From Testing to Debugging
MistakeFailureFault
TopicWhat to remember
TestingMain purpose is to find failures (symptoms: observed wrong behavior/results). Output often includes failing tests.
DebuggingGiven a failure, the purpose is to locate and fix the fault (root cause), then produce code changes that prevent future failures.
Mistake → Fault → FailureDeveloper mistakes can introduce a fault (defect/bug), which can later manifest as a failure during execution.
Program debugging
Two major steps are: (1) locate fault and (2) fix fault. The root cause may also involve configuration, data, or hardware context.
Why Re-test After a Fix
Re-testRegression check
  • Confirm the original fault is fully fixed.
  • Make sure the fix does not introduce new faults.
Debugging Process — TRAFFIC
TrackReproduceAutomateFind OriginsFocusIsolateCorrect
StepAction
TrackEnter a problem (bug) report in a tracker (e.g., Bugzilla) with details for reproduction.
ReproduceFollow the report to reproduce the same failure reliably.
AutomateWrite system/unit tests so the failure can be reproduced automatically.
Find OriginsTrace backward from a known infection/failure point along the infection chain.
FocusInspect likely wrong locations (fault-prone history, explicit assertion violations).
IsolateNarrow down candidate causes using observation and experiments.
CorrectApply and validate the actual fix at the faulty location.

7 Key Elements of Tracking & Reproducing
DescribeIsolateGeneralizeSummarizeCondenseDisambiguateNeutralize
#ElementWhat to remember
1DescribeProvide specific steps for others to reproduce the problem.
2IsolateFind the relevant parts of inputs/steps — narrow conditions until the problem disappears when you remove them.
3GeneralizeBroaden to find other inputs/steps that also reproduce the same problem (e.g. also happens with Wingdings).
4SummarizeCompress all findings into one succinct sentence (e.g. "Arial, Wingdings, and Symbol fonts corrupt new files.").
5CondenseShorten verbose sentences while preserving full meaning.
6DisambiguateRemove ambiguities — add clarifying phrases so the description is unambiguous.
7NeutralizeRemove negative/emotional language; replace with objective, factual descriptions.
Exam tip
Isolate and Generalize can be used together — they are not mutually exclusive. Quiz Q3 "do not happen together" → False.

Binary Search to Simplify Failing Input
SplitDiscard Halflog₂(n)
StepWhat happens
SplitDivide the current failing input into two halves.
Try first halfIf still failing → keep it, repeat. If passing → discard it, try the other half.
RepeatKeep halving until you cannot reduce further.
CostNeeds about log₂(n) tries for an input with n lines.
Exam tip
Q1: failing input with 1000 lines → log₂(1000) tries. The other options (logₑ, 2^1000, e^1000) are all wrong.
Delta Debugging — When Binary Search Gets Stuck
Few Large → More SmallGranularityAlternatives
Binary search gets stuck when both halves pass independently. Delta debugging handles this by refining the granularity of changes.
PhaseWhat to do
StartTry few, large changes (e.g. remove halves) to produce alternative smaller inputs.
Both pass?Increase granularity: split into more, smaller changes (quarters, eighths, …).
Found failing?Adopt the new smaller failing input and restart with it.
StopWhen no smaller alternative produces a failing input — that is the minimum failing input.
Exam tip
Q3: delta debugging starts with few, large changes → more, small changes. The other orders are all wrong.
Relationship
Q2: delta debugging is used to further simplify after binary search can no longer make progress → True.

Three Main Activities in Fault Localization
TracingBacktrackingCause EliminationCoverage-based
ActivityWhat it meansTechniques
TracingWatch execution to collect runtime info (variable values, branch taken) and determine when it goes wrong.Print statements, single-stepping, logging (log4j), breakpoints, assertions/watchpoints.
BacktrackingStart from the observed failure and follow the error propagation chain backward to find the first error / fault location.Add print statements at hypothesized earlier points; repeatedly re-run and trace back further.
Cause eliminationList all possible causes of the failure and check each one by examining the program and running experiments.Use a debugger to force values at runtime; formulate hypotheses and test them.
Exam tip
Q1 "following error propagation chain" → Backtracking. Q2 "breakpoints / assertions / print statements" → Tracing. Q3 "check every possible cause" → Cause elimination.
Expertise Needed for Cause Elimination
Program KnowledgeCommon FaultsMental Execution
  • Must understand the model / expected behavior of the program — can mentally simulate execution without running it.
  • Must know common developer faults for the language/platform, e.g.:
    • C/C++: null pointer dereference, pointer errors, memory leaks.
    • File I/O: not closing a file after opening and reading.
    • OOP: not initializing objects correctly; invoking a method on the wrong object.
Exam tip
Q4 "cause elimination requires expertise about the program and common developer faults" → True.

Assertions in Debugging vs JUnit
Program AssertionJUnit AssertionDifferent
TypeWherePurposeExamples
Program assertionInside production / debug codeDetect errors close to where they occur — easier to trace back to the fault.assert !isEmpty() : "Stack is empty"
JUnit assertionInside test codeCheck observable behavior from outside the program (black-box style).assertEquals, assertTrue
Program assertions detect errors closer to the fault location; JUnit assertions detect propagated errors at the failure location — so the two serve different roles.
Java assert Syntax & Enable / Disable
assert exp1: exp2-eaProduction
SyntaxMeaning
assert exp1;If exp1 is false at this point, throw AssertionError.
assert exp1 : exp2;exp2 is the argument (message) of the exception thrown when exp1 fails — used for diagnosis.
  • Enable assertions in JVM: -ea[:package|:ClassName]
  • Disable assertions in JVM: -da[:package|:ClassName]
  • Assertions are typically disabled in production to avoid performance overhead — especially for high-performance systems.
Key distinction
exp2 is NOT a second condition — it is the exception message/argument shown when the assertion fails.

Exam tip — Q1
"Assertions in debugging are different from those in testing" → True. They differ in location, purpose, and syntax.
Should Assertions Be in Production Code?
NoPerformance Overhead-ea / -da
  • Assertions add runtime overhead — each check costs execution time.
  • High-performance production systems typically disable assertions (-da) to avoid this cost.
  • Assertions in production code should not be relied upon as normal control-flow logic.
Exam tip — Q2
"Should assertions be part of production code?" → No.

Types of Assertions — Design by Contract
PreconditionPostconditionClass InvariantLoop Invariant
TypeWhen trueWhat to remember
PreconditionBefore a method executesConditions the method requires to work correctly; guards against illegal inputs.
PostconditionAfter a method executes (given precondition held)Describes what the method guarantees on exit. Often hard to specify in practice.
Class invariantBefore and after every public method bodyActs as both precondition and postcondition for every public method in the class.
Loop invariantBefore loop, after loop, every iterationA property that holds at all three points throughout the loop's lifecycle.
Exam tip — Q1
"Assertion that must be True after a method execution"
→ Postcondition (Precondition = before; Checkpoint = inside program).
Invariants — Always True
Always TrueClass InvariantLoop Invariant
An invariant is an assertion that should be always true at the location where it is specified — both types hold across all relevant execution points.
  • Class invariant
    example: Math.MIN_INT ≤ lower < upper ≤ Math.MAX_INT
    — must hold at the start and end of every public method (setLower, setUpper).
  • Loop invariant
    example for summing 1…n: s == sumFrom(i+1, n)
    — holds before the loop, before each iteration, and after the loop exits.
Practical Usage of Contract Assertions
Write PreconditionsPostconditions HardUnit Tests Instead
TypePractice
PreconditionsEncouraged — relatively easy to write; improves robustness against illegal inputs.
Class invariantsEncouraged when not too complicated to specify.
PostconditionsOften skipped — too complex; developers write unit tests (parameterized tests) instead to cover specific scenarios.
Failure, Fault & the Debugging Chain
MistakeFaultFailureDebuggingTesting
TermDefinition & exam angle
MistakeHuman error made by the developer — introduces a fault.
FaultA defect / bug in the code — the root cause. “Cause of bugs” = Fault.
FailureIncorrect behavior observable at runtime — the symptom. “Symptoms of bugs” = Failure.
TestingAims to find failures. Testing ≠ Debugging (common False Q).
DebuggingGiven a failure, aims to locate and fix the fault. Purpose = fix faults, not “fix failures.”
Two major steps of debugging: (1) locate the fault, (2) fix the fault. After fixing, always re-test: confirm the fix works and check no new faults were introduced.
Exam tip
Debugging can introduce new bugs (regression) → True. If a bug cannot be reproduced it does not mean it no longer exists → False.
