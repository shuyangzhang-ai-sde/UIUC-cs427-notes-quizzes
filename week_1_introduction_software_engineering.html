Software Development Life Cycle (SDLC)
SDLC Requirements Design Implementation Testing Maintenance
The software development life cycle (SDLC) is the end-to-end framework from inception through release and maintenance, with tasks and deliverables for each phase.
PhaseMain activitiesTypical artifacts
RequirementsElicit, analyze, and document needsSoftware Requirements Specification (SRS)
DesignArchitecture, modules, interfacesDesign documents, UML diagrams
ImplementationCoding and unit integrationSource code
TestingVerify behavior and qualityTest plans, reports, defect logs
MaintenanceFix defects, adapt to changePatches, releases
Exam tip
“Evaluation phase” is not a standard SDLC phase. It is a common distractor; the usual list is requirements, design, implementation, testing, and maintenance.

Information Hiding
Information Hiding Encapsulation Interface Modularity
Hide a component’s internal implementation and expose only its interface. When internals change, external code does not need to change.
  • Change internals → external code stays unaffected
  • Improves modularity and lowers coupling
  • Core idea behind encapsulation in OOP
ConceptFocus
Information HidingHide internals; expose the interface
Divide and ConquerSplit big problems into smaller ones
InheritanceSubclass reuses parent fields/methods (OOP)

Brooks’s Law: adding people can slow you down
Brooks’s Law Software Crisis Communication Overhead Onboarding
“Adding manpower to a late software project makes it later.” — Fred Brooks, The Mythical Man-Month
  • New hires need onboarding time and produce little at first
  • More people → communication paths grow → communication overhead spikes
  • Veterans mentor newcomers → overall throughput drops
In short
Adding people does not fix the software crisis; it often makes things worse.
Project Stakeholders
Stakeholder Customer User Developer Tester Designer
A stakeholder is anyone with an interest in or impact on the project—not only developers.
RoleTypical part in the project
CustomerPays for or commissions the work
UserUses the software day to day (may differ from the customer)
DesignerSystem and UI/UX design
TesterValidates quality and correctness
Support engineerDeployment, operations, user support
Project managerSchedule, resources, risk
Users are not always customers
  • Customer: who pays (e.g., procurement or IT)
  • User: who operates the software (e.g., staff)
  • Often different in B2B; sometimes the same in consumer apps
Example
The IT department buys a CRM (customer); the sales team uses it (users).
Software Engineering vs. Programming
Software Engineering Programming Multi-person Multi-version Long-lived Systems
DimensionProgrammingSoftware engineering
ScaleIndividual or small scriptsTeam-based development
VersionsOften one-off or single releaseMany releases; evolution over time
LifetimeShort-term, ad hocLong-term maintenance and change
FocusGetting code to runProcess, design, quality, collaboration
OutcomeA working programA sustainable, maintainable system
  • Software engineering includes coding plus requirements, design, testing, maintenance, and more.
  • The hard part is keeping quality and consistency under teamwork and long-term change.
  • It is “how to build software well at scale,” not only “how to write code.”
Key trait of software engineers
Beyond technical skill, you need the ability to collaborate effectively in a team—that is what distinguishes engineers from programmers who only write code in isolation.
Competing goals of a software project
Time Cost Quality Iron Triangle Trade-off
Time, cost, and quality trade off—you rarely optimize all three. This is the Iron Triangle.
Time
Schedule and calendar time for development
Cost
What the customer pays the team
Quality
Functional completeness and reliability
  • Faster delivery → spend more or cut quality
  • Lower cost → take longer or cut quality
  • Higher quality → more time or more money
