Week 3 · Requirements Engineering

What is requirements engineering?
Requirements engineering Real-world goals Constraints Stakeholders
Requirements engineering is about the real-world goals the system should support, the functions it must provide, and the constraints on the software — plus discovering and aligning what stakeholders expect.
  • Helps engineers understand and refine requirements over time
  • Surfaces stakeholder needs and calibrates expectations
  • Does not specify how the system will be built — that belongs to design
Exam tip
Saying RE is “concerned with designing a software system with respect to requirements” is wrong. RE is about discovering and defining requirements, not choosing implementations.
Why write requirements documents?
Manager Developer Tester Support engineer
Different roles read the same specification for different legitimate reasons:
RoleTypical useValid reason to rely on the spec?
ManagerPlan resources and track delivery against scope✓ Yes
DeveloperImplement behavior that matches the specification✓ Yes
TesterCheck the product against agreed behavior✓ Yes
Support engineerDecide whether to retire or discontinue a system✗ No
Exam tip
Using the requirements spec mainly to decide whether to discontinue a system is not a standard reason for writing requirements. Sunsetting is a business or operations decision, not what the SRS is for.

You can document requirements with many notations, but a test plan is not a requirements-specification notation.
UML
Models structure and behavior (classes, interactions, …)
User story
Lightweight, user-centric statements (common in agile)
Data flow diagram
Shows how data moves through processes
Test plan ✗
Belongs to testing, not to stating requirements
Functional vs. non-functional requirements
Functional requirement Non-functional requirement Behavior Quality attribute
Requirements split into what the system does (functional) and how well it does it — quality, performance, and constraints (non-functional).
FunctionalNon-functional
FocusConcrete behavior or services the system providesQualities, performance bounds, constraints
Typical form“The system shall …” (send, record, display, …)“Within N seconds …”, “99.9% available …”, “user-friendly …” (must be made measurable)
Keywordsshall send, shall record, shall assign, …within N seconds, within N minutes, …

Worked examples

StatementKindWhy
A text message shall be sent when a customer signs up.FunctionalSpecifies a system action in response to an event
The table's full contents shall be displayed within 3 seconds.Non-functionalTime bound — a performance requirement
A new user shall finish ordering within 10 minutes.Non-functionalTime bound on the interaction — often treated as usability / performance
Every invoice shall be assigned a unique identifier.FunctionalData-handling rule the system must enforce
The server code shall be written in C.ConstraintImplementation or technology constraint (often classed with non-functional or separate constraints)
Heuristic
Phrases like within N seconds, user-friendly, or available 99.9% usually signal non-functional requirements (once refined to be testable). Verbs that name system actions (send, record, assign) usually signal functional requirements.
Mandatory vs. optional: shall, should, will
shall should will Mandatory Optional
Verb choice signals how binding a statement is:
TermForceTypical use
shallMandatoryMust be satisfied; baseline for acceptance
shouldRecommendedDesirable but negotiable if trade-offs arise
willStatement of fact / intentDescribes expected behavior without the same normative weight as shall in many standards
Example
“The driver’s door ____ be closed once the car is moving.” → shall — safety-critical, mandatory behavior.
Prescriptive vs. descriptive statements
Descriptive Prescriptive Domain fact
Requirements mixes two kinds of sentence:
KindMeaningSignals
DescriptiveStates a domain fact that holds in the world (physics, policy, …)No normative modal like shall; often background for the model
PrescriptiveStates what the system must or should doOften uses shall / should; binds the system under development
Examples
  • “The same person cannot be in two buildings at once.” → Descriptive — a physical-world fact, not a system obligation.
  • “A user shall not use the same registration code more than once.” → Prescriptive — a rule the system must enforce.
Requirements elicitation
Elicitation Stakeholder expectations Interview Brainstorming Survey
Elicitation is how you discover what stakeholders need and negotiate those expectations. It is not the same as telling developers how to build the system, and not the same as formal product verification.

Common techniques

TechniqueWhat it isElicitation?
Questionnaires, surveys, interviewsStructured or semi-structured input from stakeholders✓ Yes
Brainstorming and focus groupsGroup sessions to surface needs and priorities✓ Yes
Early stakeholder feedbackReactions to prototypes, demos, or early builds✓ Yes
User acceptance testing (UAT)Formal check that the delivered system meets agreed criteria✗ No
Exam tip
UAT belongs to testing: it validates an implementation against requirements. Elicitation happens earlier, to learn what the requirements should be — not to certify the final build.
Use-case artifacts — three parts
Use case Actor Use case diagram Test case
A conventional use-case specification rests on three pieces. Test cases are not one of them — they belong to testing.
Use case
Text describing the sequence of events when the system responds to a stimulus
Use case diagram
Shows actors and use cases and how they relate
Actors
External entities (people or systems) that interact with the system under design
Test cases ✗
Verification artifacts, not part of the three-part use-case model
Actors live outside the system
Actor Input Output External
An actor is any external entity that interacts with the system: a person, another system, a hardware device, …
  • Actors may supply inputs to the system
  • Actors may receive outputs from the system
  • They may both send and receive in the same scenario
  • Actors are drawn outside the system boundary — they are not internal components
Common mistake
“Actors are inside the system” is false: by definition they sit outside the system you are specifying.
Four levels of use-case detail
Actor–goal list Use case brief Casual use case Fully dressed use case
FormDetailMain use
Actor–goal listMinimal — list of goals per actorScoping what to design
Use case briefShort narrativeQuick sketch of the main success path
Casual use caseParagraph-style proseMain success plus some alternatives and failures
Fully dressed use caseStructured sections (preconditions, steps, extensions, …)Full coverage of scenarios and exceptions
Casual vs. fully dressed
Both center on main success scenarios and usually cover failures and alternatives; the difference is structure — casual prose vs. labeled sections.

Neither format tells developers how to code the feature; they state what should happen.
Exam tip
Claiming that casual and fully dressed use cases exist to tell developers how to implement requirements is false. They specify behavior, not implementation.
What a good use case does
Event sequence End-to-end behavior Readable Multiple actors
Strong use cases tend to:
  • Describe the sequence of events when the system responds to an actor
  • Cover one coherent slice of functionality from trigger to completion
  • Stay readable — short but still a complete “story”
  • Allow more than one actor when the scenario truly involves several
Common mistake
“A use case cannot be associated with more than one actor” is false — multiple actors may share a use case when they all participate in that behavior.
Common flaws in requirement statements
Unmeasurability Contradiction Poor structuring Inadequacy
Sound requirements are testable, consistent, well structured, and complete enough. Four recurring problem types:
FlawWhat it meansTypical symptom
UnmeasurabilityNo objective way to verify compliance“The system shall be user-friendly.” / “The system shall be fast.”
ContradictionTwo statements cannot both be satisfiedConflicting limits on the same behavior
Poor structuringHard to read or traceSeveral requirements crammed into one sentence
InadequacyImportant cases left unstatedMissing boundaries, errors, or edge behavior
Example
“The book-order process shall be user-friendly.” → classic unmeasurability — replace with something testable, e.g. “shall be completable in at most three clicks from the catalog.”
