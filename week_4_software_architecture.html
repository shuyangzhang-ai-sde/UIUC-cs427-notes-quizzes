Abstraction — Unix file descriptors
Abstraction File descriptor Unix Interface
Abstraction means exposing a small interface while hiding how things work underneath.

In Unix, a file descriptor is the uniform handle for I/O: a process reads and writes using that integer without knowing the file’s real path or storage location.
  • Once a process holds a file descriptor (an integer handle), it can read and write through it.
  • Whether the backing resource is a regular file, socket, or pipe is transparent to the process.
  • That is the point: callers stay unaware of implementation details.
Exam tip
False. The claim that a process must know a file’s actual name to use it is wrong under the file-descriptor abstraction: access goes through the descriptor, not the string path.

Pipe-and-filter architecture
Pipe-and-filter Filter Pipe Data flow Unix
Data moves through a chain of filters. Each filter transforms the stream and hands it to the next stage via a pipe, until a final output is produced.
  • Each filter is an independent stage and does not depend on knowing how other filters work.
  • Filters do not share mutable state (they are effectively stateless with respect to each other).
  • Data flows in one direction.
  • Filters may be implemented in parallel where the model allows.
  • Processing can be incremental — work as data arrives.
Classic example
Unix shell: cat file.txt | grep "error" | sort | uniq
Each command acts as a filter; | is the pipe between stages.
Exam tip (Q5)
Not true: “Filters know each other and share state.” In pipe-and-filter, stages stay loosely coupled and do not share state the way collaborating objects in a layer might.

Sensor data passes through a pipeline of transforms ending in visualization — that is sequential dataflow with independent stages, i.e. pipe-and-filter, not primarily a layered style.
Components vs. connectors
Component Connector Interaction Rules Encapsulation
ComponentConnector
What it isA unit that encapsulates computation and/or dataThe mechanism that mediates interaction between components
RoleImplements behavior and may hold stateDefines how components communicate and the rules of those exchanges
ExamplesService, module, filter, layerPipe, event bus, RPC, protocol stack
Connectors cover:
  • Interactions among components
  • Rules that govern those interactions
  • Not “encapsulates computation and/or data” — that is a component concern.
Exam tip (Q6)
“Encapsulates computation and/or data” describes a component, not a connector. Connectors coordinate communication; they do not replace component encapsulation.
MVC — model, view, controller
Model View Controller Separation of concerns Multiple views
Model
Domain data and rules; no UI detail
View
Presentation — what the user sees
Controller
Handles input (e.g. gestures and commands) and coordinates model and view
  • The controller receives user-gesture and input events (clicks, typing, taps).
  • The model does not depend on a specific view; you can change views without rewriting core logic.
  • One model can drive multiple views — the same data as a line chart, pie chart, or table.
Exam tip (Q8)
Different charts over the same data illustrate MVC’s multiple views: views stay pluggable because they render from a shared model without duplicating business rules.
Exam tip (Q7)
User-gesture events go to the controller, not the view (the view renders output).
Software modularity
Modularity Independence Reusability Robustness Compilation
Decompose the system into separate modules, each with a clear boundary; modules interact through explicit interfaces.
Why modularity helps:
  • Faults tend to stay localized — modules can be more robust to change.
  • Modules can reuse one another (one module may call or compose others).
  • Modules can be compiled separately and shipped in libraries.
  • Designs aim for loose coupling: modules are mostly independent, not tightly dependent on everything else.
Exam tip (Q9)
Incorrect claim: “Modules are mostly dependent.” Modularity seeks independence and manageable coupling — not pervasive mutual dependence.
Conway’s law
Conway's Law Team structure System architecture Communication
“Organizations which design systems are constrained to produce designs which are copies of the communication structures of those organizations.”

In practice: the system’s structure often mirrors how teams talk and coordinate.
  • Compiler front-end team → owns the front-end subsystem
  • Compiler back-end team → owns the back-end subsystem
  • Team boundaries tend to line up with module boundaries

Separate teams owning compiler front end vs. back end — matching org splits to architectural splits — is a textbook illustration of Conway’s law.
