Week 7 — Software Testing

What testing can and cannot do
Testing Code Quality Fault Detection Limitation
Testing helps you find defects, improve quality, and document behavior, but it has a hard limit: it cannot prove the program is fully correct.
Claim about testingWhat it meansValid?
Improve code qualityFind and fix defects so quality rises.✓
Spot code faultsReveal bugs and wrong logic.✓
Document codeTest cases act as executable specs of system behavior.✓
Guarantee fully correct behaviorThe input space is huge; you cannot exhaust every case.✗
Exam tip
Testing does not guarantee the code is “fully correct.” That line is a classic distractor. Passing every test still leaves room for untested paths.
When to test and how much test code
TDD Test-first Test Code Volume UI Testing
You can write tests at any point in the lifecycle, and the test codebase can be larger than the production codebase.
  • Before — Write tests first (TDD/BDD) to drive design.
  • During — Add tests alongside features so new code stays covered.
  • After — Backfill tests for regression fixes and legacy code.
  • In mature projects, test code can outgrow source code; that is normal.
  • UIs are testable too; automate UI checks once layouts and flows stabilize.
Who writes tests
Testing is not QA-only. Developers, domain experts, and users can all author test cases—more perspectives means fewer blind spots.
Developer bias
Developer Bias QA Independent Testing
Developers who test their own code often show cognitive bias: they exercise paths they “know” work and miss edge cases.
  • Deep understanding of the code does not mean you will find every fault.
  • It is easy to assume your implementation is right and skip adversarial cases.
  • Independent reviewers and diverse authors catch what the original author misses.
  • QA engineers specialize in testing, but they are not the only people who should write cases.

Exam tip
“Test Implementation” is not a named activity in this model. It is a common distractor. Memorize: Design → Automation → Execution → Evaluation.
Manual vs automated testing
Automated Testing Manual Testing Regression Testing
TopicManual testingAutomated testing
Typical useExploration, one-off checks, early UI mockups.Regression suites and anything run on every build.
Finding bugsStrong for creative exploration and edge cases.Does not automatically beat manual testing in bug count.
MaintenanceLow—no scripts to update.Higher—tests must track production changes.
UI timingManual checks work while designs churn.Automate after the UI stabilizes.
Reality check
Automation is not “more bugs by default.” One-off or rarely repeated checks are often cheaper to run manually; scripting them can waste effort.
Regression
When the same regression battery must run often, automation pays off—that is where it earns its keep.
Test cases as living documentation
Test Cases Living Documentation Requirements Understanding
Authoring tests forces you to clarify requirements. Those cases also document how the system is supposed to behave.
  • Tests are not an afterthought—they state expected behavior.
  • Choosing inputs, outputs, and boundaries is requirements work in disguise.
  • Teammates can read tests as specs.
  • Tests need not be code; manual plans still count.

Exam tip
White-box work is not “easy”—it demands solid coding skill. Black-box testing is comparatively weak on extra-logic faults because you never inspect the implementation.
Why black-box testing cannot be exhaustive
Exhaustive Testing Input Space Infeasible
The practical obstacle is an enormous input space: you cannot run every possible input, so you sample intelligently.
  • Even tiny routines can admit astronomical combinations.
  • Use equivalence class partitioning and boundary value analysis to shrink the suite.
  • This is the same reason testing cannot “prove full correctness.”
Black-box techniques: ECP vs BVA
Equivalence Class Partitioning Boundary Value Analysis Decision Table Testing Use Case Testing
TechniqueIdeaMnemonic
Equivalence class partitioningSplit inputs into classes that should behave alike; pick one representative per class.Partition, representative.
Boundary value analysisStress min, max, and neighbors (e.g. ±1) at class edges.Edges, ±1.
Decision table testingTabulate condition combinations and expected actions.Matrix of rules.
Use-case testingDerive scenarios from real user workflows.End-to-end stories.
How ECP and BVA pair
ECP answers “which groups of inputs matter?” BVA asks “what happens right at the edges of those groups?” Use ECP to choose classes, then BVA to harden boundaries.
White-box cost and reusable design tactics
Maintenance Cost Equivalence Class Boundary Value
White-box suites track implementation detail, so frequent refactors inflate maintenance cost.
Cross-over
Equivalence class partitioning and boundary value analysis are not black-box–only—they are general design tactics you can apply in white-box testing too.
PIE model: fault → failure chain
Execution Infection Propagation Failure Observation
PIE lists three necessary conditions for an observed failure. Drop any one, and users never see the symptom.
P — Execution
The faulty statement actually runs.
I — Infection
Program state becomes incorrect.
E — Propagation
That bad state reaches an observable output.
Exam tip
All three are necessary. Wording like “the infected state must not propagate” is wrong—without propagation, there is no visible failure.
Error, fault, and failure
Error Fault Failure
TermMeaningLevel
ErrorA human mistake while designing or coding.People / process
FaultThe defect frozen into the artifact—buggy code or design.Static artifact
FailureWrong external behavior you can observe at runtime.Dynamic / observable
Walk-through
You meant to call Math.log(5) but coded Math.pow(5, 3), returning 125. The slip while typing is an error; the bad expression in source is the fault; returning 125 instead of ln(5) at run time is the failure.
Code coverage metrics
Statement Coverage Branch Coverage Method Coverage Parameter Coverage ✗
Coverage shows how much of the program structure your tests exercise—useful, but not a proof of correctness.
MetricWhat it checksStandard?
Statement coverageEach executable statement ran at least once.✓ Yes
Branch coverageEach decision’s true/false outcomes.✓ Yes
Method coverageEach method or function invoked.✓ Yes
Condition coverageEach boolean sub-condition in compound predicates.✓ Yes
“Parameter coverage”Not defined in the usual taxonomy.✗ No
Exam tip
Parameter coverage is not a mainstream coverage dimension. Recognized metrics tie to executable structure—statements, branches, methods, conditions—not to parameters by themselves.
