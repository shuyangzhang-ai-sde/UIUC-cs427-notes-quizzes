Week 8 — Software Testing with JUnit

JUnit Framework
JUnit 5AssertionsAssumptionsAnnotationsTest SuitesAutomated Tests
  • Open source Java unit testing framework (junit.org) for writing and running repeatable automated tests
  • The latest version is JUnit 5, requiring Java 8 (or higher) at runtime; can still test code compiled with previous JDK versions
  • Main JUnit features:
    • Assertions for checking expected results
    • Assumptions for filtering out tests
    • Annotations for configuring tests and extending the framework — disabling tests, conditional test execution, displaying names, tagging and filtering
    • Test suites for organizing and running tests
    • Console launcher / test engines for executing tests

Example JUnit Test Case
@Test@DisplayNameassertEqualsassertFalse
Class under test: java.util.Stack — Test classes/methods don't need to be public.
import java.util.Stack;
import org.junit.jupiter.api.*;
import static org.junit.jupiter.api.Assertions.*;

class TestingAStackDemo
{
    @Test
    @DisplayName("Push an element into empty stack")
    void testPushOneElem()
    {
        String element1 = "element1";
        Stack<Object> stack = new Stack<>();
        stack.push(element1);
        assertEquals(element1, stack.peek());
        assertFalse(stack.isEmpty());
    }
}

Anti-Pattern: Tests Combined
Anti-PatternFail FastState-modifying
Avoid combining multiple unit tests into a single big test. If an earlier assertion fails, the whole test stops — later assertions never run, limiting debugging information.
class TestingAStackDemo
{
  Stack<Object> stack;
  String element1 = "element1";
  String element2 = "element2";
  @BeforeEach
  void createNewStack() {
    stack = new Stack<>();
  }

  @Test
  void testAddOneAndTwoElems()
  {
    stack.push(element1);
    assertEquals(element1,
      stack.peek());
    stack.push(element2);
    assertEquals(element2,
      stack.peek());
  }
  @AfterEach
  void createNewStack() {
    assertFalse(
      stack.isEmpty());
  }
}
Q5 — Exam tip
A big test is better than several small tests. → False. Avoid placing state-modifying calls between assertions; break combined tests into separate small tests.
Assertions — assertEquals & assertSame
assertEqualsassertSamedeltaorg.junit.jupiter.api.Assertions
Static methods defined in org.junit.jupiter.api.Assertions. Assertions appear at the end of a test method; if one fails, execution stops and the following assertions do not run.
SignatureNotes
assertEquals(byte expected, byte actual)Exact equality for primitives
assertEquals(char expected, char actual)Exact equality for primitives
assertEquals(int expected, int actual)Exact equality for primitives
assertEquals(long expected, long actual)Exact equality for primitives
assertEquals(short expected, short actual)Exact equality for primitives
assertEquals(double expected, double actual, double delta)delta required — asserts equal within the given delta
assertEquals(float expected, float actual, float delta)delta required — asserts equal within the given delta
assertEquals(Object expected, Object actual)Uses equals() — value equality
Only double and float have the delta overload. This is because floating-point arithmetic can introduce rounding errors, so two values that should be equal may not compare as exactly equal; delta specifies the allowable difference.
SignatureNotes
assertSame(Object expected, Object actual)Reference equality — asserts both refer to the same object
assertEquals(Object, Object) calls equals() on the objects — it checks value equality.
assertSame checks reference identity: it asserts that the expected and actual refer to the exact same object in memory.
The two behave identically only when equals() has not been overridden, because the default equals() is itself reference-based.
assertAll
assertAllExecutableGrouped Assertions
static void assertAll(String heading, Executable… executables)
Asserts that all supplied executables do not throw exceptions. Every executable is run to completion regardless of whether earlier ones fail — this is the key difference from writing assertions sequentially, where the first failure immediately stops the test. With assertAll, all failures are collected and reported together under the given heading.
@Test
void groupedAssertions() {
    // In a grouped assertion all assertions are executed, and any
    // failures will be reported together.
    assertAll("person",
        () -> assertEquals("John", person.getFirstName()),
        () -> assertEquals("Doe",  person.getLastName())
    );
}
assertThrows
assertThrowsExecutableexpectedTypeException Testing
static <T extends Throwable> T assertThrows(Class<T> expectedType, Executable executable)
Asserts that execution of the supplied executable throws an exception of the expectedType. The thrown exception is returned by the method, so it can be further inspected — for example, by checking its message with assertEquals.
@Test
void exceptionTesting() {
    Throwable exception = assertThrows(
        IllegalArgumentException.class, () -> {
            throw new IllegalArgumentException("a message");
        });
    assertEquals("a message", exception.getMessage());
}
assumeTrue & assumeFalse
assumeTrueassumeFalseorg.junit.jupiter.api.Assumptions
Static methods in org.junit.jupiter.api.Assumptions. If the assumption fails, the rest of the test is skipped (aborted), not failed. Common use: guard a test that only makes sense in a specific environment.
MethodCondition trueCondition false
assumeTrue(boolean)Continue rest of testSkip (abort) rest of test
assumeFalse(boolean)Skip (abort) rest of testContinue rest of test
 
@Test
void onlyOnCI() {
    assumeTrue("CI".equals(System.getenv("ENV")));
    // only executes when ENV=CI; skipped otherwise
    assertEquals(expected, actual);
}
assumingThat
assumingThatExecutable
assumingThat(boolean assumption, Executable executable) — argument order: Boolean first, Executable second.

If assumption is true: the executable block runs.
If assumption is false: the executable block is skipped.

Either way, code after assumingThat always continues — unlike assumeTrue/assumeFalse which abort the entire remainder of the test.
@Test
void test() {
    assumingThat(
        "CI".equals(System.getenv("ENV")),
        () -> { assertEquals(2, 2); }  // skipped when ENV != CI
    );
    System.out.print("2");  // always executes
}
Parameterized Tests
@ParameterizedTestparametersJUnit 5
  • A parameterized test is a unit test with parameters; conventional unit tests have no parameters
  • The same test method is invoked multiple times, once per argument value from the source
  • Must annotate the test method with @ParameterizedTest (not @Test)

@MethodSource
@MethodSourcestatic factoryStreamArguments
  • Specifies one or more factory methods (on the test class or an external class) that supply arguments
  • Factory method must be static (unless @TestInstance(PER_CLASS) is used)
  • Return types: Stream, Iterator, Iterable, or array
  • For multiple parameters: return Stream<Arguments>, wrap each set with Arguments.of(…)
  • External factory: use fully qualified name "com.example.Class#method"
@ParameterizedTest
@MethodSource("stringProvider")
void test(String arg) { ... }

static Stream<String> stringProvider() {
    return Stream.of("foo", "bar");
}

// multiple parameters
static Stream<Arguments> multiProvider() {
    return Stream.of(Arguments.of("Alice", 1, true));
}

Explicit Conversion: @ConvertWith
@ConvertWithExplicit conversionCustom converter
  • Annotate the parameter with @ConvertWith(MyConverter.class)
  • JUnit uses your converter class to convert the source value to the target type
  • Unlike fallback, this is explicitly declared by the developer
@ParameterizedTest
@ValueSource(strings = {"1, 2", "1, 2, 3"})
void test(@ConvertWith(CSVtoArray.class) String... arg) { ... }

public static class CSVtoArray extends SimpleArgumentConverter {
    @Override
    protected Object convert(Object source, Class<?> targetType) {
        return ((String) source).split("\\s*,\\s*");
    }
}

ArgumentsAccessor & @AggregateWith
ArgumentsAccessor@AggregateWith
  • ArgumentsAccessor: use as a parameter to access individual CSV cells inside the test body
    • accessor.getString(0), accessor.get(2, Gender.class)
    • Construct the target object (e.g. new Person(…)) inside the test method

4A Pattern: Assume, Arrange, Act, Assert
AssumeArrangeActAssert
PhasePurposeExample line
AssumeFilter invalid inputs; if assumption fails, test is skipped (not failed)assumeTrue(stk != null)
ArrangeBring object to the right state; cache values for later assertionsint oldSize = stk.size()
ActInvoke the method under teststk.push(elem)
AssertVerify expected behavior; may use cached values from ArrangeassertEquals(oldSize + 1, stk.size())
 
@ParameterizedTest
@ValueSource(...)
void PT1(int elem, Stack<Integer> stk) {
    assumeTrue(stk != null);                    // Assume
    int oldSize = stk.size();                   // Arrange
    stk.push(elem);                             // Act
    assertEquals(oldSize + 1, stk.size());      // Assert
}

Property-First vs Assumption-First Guidelines
Property-firstAssumption-firstModifiersObservers
GuidelineStarting pointSteps
Property-firstGeneral properties of the class1. Classify methods into modifiers (change state: constructor, insert, remove) and observers (return info: member, isEmpty)
2. Pairwise-combine: each observer with a modifier; each modifier with another modifier
3. Add assumptions at the start of each pair-sequence
Assumption-firstSpecific observer-return combinations1. Enumerate all feasible combinations of observer return values (e.g. isMember true/false, isEmpty true/false)
2. For each combination, formulate the properties that hold under those conditions
3. Continue with property-first steps
Quiz Q4
"Formulate properties specific to each feasible combination of observers’ returns as assumptions" belongs to Assumption-first, NOT Property-first. → This option does not belong to the Property-first guideline.
4 Steps of Test Generalization
ParameterizeGeneralize Test OracleAdd AssumptionsCross Test Generalize
StepWhat you doExample
1. ParameterizePromote primitive values and non-primitive objects (e.g. receiver objects) from the test body to parameters of the parameterized testint elem = 1 becomes a parameter int elem; empty Stack becomes parameter Stack stk
2. Generalize test oracleReplace constant values in assertions with generalized expressions that remain valid for any parameter valueassertEquals(1, stk.size()) becomes assertEquals(oldSize + 1, stk.size())
3. Add assumptionsAdd assumeTrue / assumeFalse at the start to prevent illegal parameter values from causing errorsassumeTrue(stk != null) filters out null references that would cause exceptions
4. Cross test generalizeMerge or generalize further across multiple parameterized tests to produce a single, more general testTwo tests pushing 1 element each merged into one test iterating over an int array and checking stk.size() == arr.length
Quiz Q1 — Q5
NOT a step: "Classify methods" (that belongs to the Property-first guideline, not test generalization).
Q2 promoting primitives as arguments → Parameterize.
Q3 replacing constants in assertions → Generalize test oracle.
Q4 adding assumptions for illegal values → Add assumptions.
Q5 generalizing further across multiple tests → Cross test generalize.

What software testing is for
TestingFunctional RequirementsVerification
Software testing runs test cases, compares actual results to expectations, and finds defects—checking that behavior matches what was specified.
Core idea
Testing is the process of checking that the software satisfies its functional requirements.
Small tests vs. big tests
Test GranularityAssertionFail Fast
In one large test, if an early assertion fails, later assertions often never run, which makes diagnosis harder. Several smaller tests fail independently and are easier to maintain.
StyleWhat happensVerdict
Big testAn early failure usually stops the rest of the assertions from running.Avoid
Small testsEach test runs on its own; one failure does not block the others.Prefer
Exam tip
Why “one big test is worse than several small tests”: when an assertion fails first, the following assertions may not execute at all.
